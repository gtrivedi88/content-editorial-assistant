= API reference

[NOTE]
====
**This documentation is for developers** who want to integrate with the Content Editorial Assistant API or build automation workflows.
====

== Overview

Content Editorial Assistant provides a REST API under `/api/v1/` and WebSocket events via Socket.IO for real-time analysis progress. All successful JSON responses include `"success": true` at the top level.

== Base configuration

=== Base URL

[source]
----
http://localhost:8000/api/v1
----

=== Content types

* **Request**: `multipart/form-data` for file uploads, `application/json` for JSON data
* **Response**: `application/json`

=== Common headers

[source,http]
----
Content-Type: application/json
X-Request-ID: <auto-generated-or-provided>
----

Every request gets an `X-Request-ID` header (read from request or auto-generated UUID). This ID appears in all log records for distributed tracing.

== Endpoints

=== Analyze content

Perform comprehensive style analysis on text content.

[source,http]
----
POST /api/v1/analyze
----

==== Request body

[source,json]
----
{
  "content": "Your text content to analyze...",
  "content_type": "concept",
  "format_hint": "asciidoc",
  "session_id": "optional-session-id"
}
----

==== Request parameters

[cols="1,1,1,3"]
|===
|Parameter |Type |Required |Description

|`content`
|String
|Yes
|Text content to analyze (max 50,000 characters)

|`content_type`
|String
|No
|Content classification: `concept`, `procedure`, `reference`, `release_notes`, `assembly` (default: auto-detected)

|`format_hint`
|String
|No
|Document format: `asciidoc`, `markdown`, `dita`, `html`, `txt` (default: auto-detected)

|`session_id`
|String
|No
|Session identifier for WebSocket progress updates
|===

==== Response

The HTTP response returns immediately with deterministic results and `partial: true`. Full results arrive via WebSocket `analysis_complete` event.

[source,json]
----
{
  "success": true,
  "partial": true,
  "issues": [
    {
      "id": "det_0",
      "type": "passive_voice",
      "rule_name": "verbs",
      "message": "Use active voice instead of passive voice.",
      "flagged_text": "was configured",
      "sentence": "The system was configured by the administrator.",
      "suggestion": "The administrator configured the system.",
      "span": [15, 29],
      "severity": "MEDIUM",
      "category": "language_and_grammar",
      "source": "deterministic",
      "confidence": 0.95
    }
  ],
  "score": 82,
  "score_label": "Good",
  "report": {
    "summary": "...",
    "categories": {}
  },
  "session_id": "abc123",
  "stats": {
    "word_count": 250,
    "sentence_count": 15,
    "block_count": 8
  }
}
----

=== Upload document

Upload and extract text from document files.

[source,http]
----
POST /api/v1/upload
----

==== Request

`multipart/form-data` with:

[cols="1,1,1,3"]
|===
|Parameter |Type |Required |Description

|`file`
|File
|Yes
|Document file (PDF, DOCX, Markdown, AsciiDoc, DITA/XML, TXT)

|`format_hint`
|String
|No
|Override format auto-detection
|===

Max file size: 10 MB (configurable via `MAX_CONTENT_LENGTH`).

==== Example

[source,bash]
----
curl -X POST \
  -F "file=@document.adoc" \
  http://localhost:8000/api/v1/upload
----

==== Response

[source,json]
----
{
  "success": true,
  "content": "Extracted document text...",
  "format": "asciidoc",
  "filename": "document.adoc"
}
----

=== Get suggestion

Get an AI-powered rewrite suggestion for a specific issue.

[source,http]
----
POST /api/v1/suggestions
----

==== Request body

[source,json]
----
{
  "session_id": "abc123",
  "issue_id": "det_5",
  "issue": {
    "type": "passive_voice",
    "message": "Use active voice.",
    "flagged_text": "was configured",
    "sentence": "The system was configured.",
    "suggestion": "Configure the system."
  }
}
----

==== Response

[source,json]
----
{
  "success": true,
  "suggestion": "Configure the system to use the new settings.",
  "source": "llm"
}
----

The `source` field indicates where the suggestion came from: `cache` (session cache hit), `deterministic` (direct replacement), or `llm` (LLM-generated rewrite).

=== Get citation

Retrieve style guide excerpt for a rule.

[source,http]
----
GET /api/v1/citations/<rule_type>
----

==== Example

[source,bash]
----
curl http://localhost:8000/api/v1/citations/passive_voice
----

==== Response

[source,json]
----
{
  "success": true,
  "rule_type": "passive_voice",
  "topic": "Verbs",
  "page": "113-116",
  "excerpt": "Use active voice in most cases..."
}
----

=== Issue actions

==== Accept issue

[source,http]
----
POST /api/v1/issues/<id>/accept
----

==== Dismiss issue

[source,http]
----
POST /api/v1/issues/<id>/dismiss
----

==== Submit feedback

[source,http]
----
POST /api/v1/issues/<id>/feedback
----

[source,json]
----
{
  "session_id": "abc123",
  "vote": "up"
}
----

`vote` is either `"up"` (valid finding) or `"down"` (false positive). Feedback is persisted to SQLite on PVC (`/app/data/feedback.db`), falling back to in-memory if PVC is unavailable.

== Error handling

=== Error response format

All error responses include `"success": false`:

[source,json]
----
{
  "success": false,
  "error": "Error description"
}
----

=== HTTP status codes

[cols="1,3"]
|===
|Status |Description

|`400`
|Bad request — missing or invalid parameters

|`404`
|Not found — unknown endpoint or resource

|`413`
|Payload too large — file exceeds `MAX_CONTENT_LENGTH` (10 MB)

|`500`
|Internal server error
|===

Browser-generated asset requests (`favicon.ico`, `manifest.json`, `robots.txt`, etc.) are filtered from 404 logs.

== WebSocket API

CEA uses Socket.IO for real-time progress during analysis.

=== Connection

[source,javascript]
----
const socket = io('http://localhost:8000');
----

=== Events (server to client)

==== `stage_progress`

Emitted at phase boundaries and per-block during LLM analysis:

[source,json]
----
{
  "session_id": "abc123",
  "phase": "llm_granular",
  "status": "progress",
  "blocks_done": 3,
  "blocks_total": 15
}
----

Phase values: `deterministic`, `llm_granular`, `llm_global`.

Status values: `started`, `progress`, `done`.

==== `analysis_complete`

Final merged results from all three phases:

[source,json]
----
{
  "session_id": "abc123",
  "issues": [...],
  "score": 82,
  "score_label": "Good",
  "report": {...}
}
----

The UI holds all results until this event arrives. Intermediate phase events are logged but do not update the UI.

==== `deterministic_complete`

Emitted after Phase 1 (~0.5-1s) with deterministic-only results.

==== `llm_granular_complete`

Emitted after per-block LLM analysis completes.

==== `llm_global_complete`

Emitted after global document review completes.

=== Request cancellation

The frontend uses `AbortController` on fetch requests. The orchestrator checks `is_analysis_current(session_id)` between phases to stop stale analyses. Stale WebSocket events are filtered by `session_id` match on the client.

== Rate limiting

Rate limiting is disabled by default (`RATE_LIMIT_ENABLED=False`). When enabled:

[cols="1,1"]
|===
|Endpoint |Limit

|Default (all routes)
|120 requests/minute

|`/api/v1/analyze`
|30 requests/minute
|===

Configure via environment variables: `RATE_LIMIT_ENABLED`, `RATE_LIMIT_DEFAULT`, `RATE_LIMIT_ANALYZE`.

== Integration example

[source,python]
----
import requests

BASE = "http://localhost:8000/api/v1"

# Analyze content
response = requests.post(f"{BASE}/analyze", json={
    "content": "The system was configured by the administrator.",
    "content_type": "procedure"
})

data = response.json()
print(f"Score: {data['score']} ({data['score_label']})")
print(f"Issues: {len(data['issues'])}")

for issue in data["issues"]:
    print(f"  [{issue['severity']}] {issue['message']}")
    print(f"    Flagged: '{issue['flagged_text']}'")

# Get suggestion for an issue
if data["issues"]:
    issue = data["issues"][0]
    suggestion = requests.post(f"{BASE}/suggestions", json={
        "session_id": data["session_id"],
        "issue_id": issue["id"],
        "issue": issue
    })
    print(f"  Suggestion: {suggestion.json().get('suggestion')}")
----
